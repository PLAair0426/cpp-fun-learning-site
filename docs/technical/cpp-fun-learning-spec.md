# C++ 趣味学习网站产品 SPEC

## 1. 文档边界

- 产品形态、功能优先级、技术方向以 `C++ 趣味学习网站 0→1 可执行方案深度研究报告.pdf` 为准。
- 课程主题、讲解内容、示例代码、Linux 相关章节以 `C++基础课件和源代码（有Linux）.docx` 为准。
- 本 SPEC 目标是把两份材料统一成一份可直接进入设计、研发、教研和运营协同的落地文档。
- 默认首发语言为 `C++17`，首发终端为 `PC Web`，移动端以浏览内容、查看进度、签到打卡为主。

## 2. 产品定位

### 2.1 一句话定义

一个面向 C++ 初学者到进阶学习者的游戏化学习网站，用“学习地图 + 边学边练 + 在线运行/判题 + 成长系统”承接完整学习闭环。

### 2.2 核心价值

1. 把传统线性课件改造成可闯关、可交互、可追踪进度的学习路径。
2. 把课件中的讲解、示例代码、作业题、Linux 实验内容结构化落库，形成持续可运营的内容资产。
3. 把“看懂”和“会写”拆成多层练习：示例运行、输出预测、找错修复、正式提交。
4. 把 C++ 最抽象的内存、指针、引用等概念做成可视化演示，降低劝退率。

## 3. 用户与场景

### 3.1 用户角色

- 初学者：零基础或只会一点 C 语言，需要从环境、输入输出、变量、分支循环开始。
- 在校学习者：有明确考试/作业需求，希望通过题目和排行榜获得反馈。
- 转岗/提升用户：希望系统补齐 C++、STL、现代 C++、Linux 与网络基础。
- 教研/内容编辑：负责把 docx 课件转成网站内容、习题和闯关路径。
- 运营/管理员：负责发布、活动、审核、排行榜和风控。

### 3.2 核心场景

- 新用户第一次进入后，从“新手村路径”开始学，完成课时后自动解锁下一关。
- 学员在章节页边看讲解边运行示例，查看 stdout / stderr / compile output。
- 学员完成小测或编程题后，立即获得通过状态、经验值、徽章或失败原因。
- 管理员从后台发布课程、题目、测试用例和活动，记录审计日志。

## 4. 产品目标与非目标

### 4.1 12 周 MVP 目标

- 上线 1 条核心学习路径，覆盖 29 节核心课。
- 上线 100–150 道题目，其中首发目标为 120 道。
- 上线 2 个 WebAssembly 趣味演示。
- 支持注册登录、学习地图、章节学习、在线运行、正式提交、学习进度、排行榜、徽章、后台发布。
- 打通判题链路、SSE 状态推送、基础监控、备份与审计。

### 4.2 非 MVP 范围

- 社区论坛
- 多人协作 IDE
- AI 自动出题/自动批改解释

- 多语言国际化

## 6. 内容结构模型

### 6.1 层级模型

- `Path`：学习路径，例如“C++ 新手村”“内存迷宫”“Linux 实验室”。
- `Course`：某一阶段课程集合。
- `Lesson`：单节课，对应 docx 的一级主题。
- `Block`：章节内容块，支持多种渲染形态。
- `Problem`：练习题或编程题。
- `Achievement`：成就定义。

### 6.2 Lesson Block 类型

- `text`：正文讲解、概念说明、注意事项
- `code`：示例代码展示
- `quiz`：单选/判断/输出预测
- `runner`：可编辑运行器
- `challenge`：正式提交题入口
- `debug`：找错修复题
- `wasm_demo`：栈/堆、指针/引用等可视化演示
- `summary`：本节总结

### 6.3 单节课标准模板

每节课统一拆成以下结构：

1. 学习目标
2. 先修提示
3. 核心概念讲解
4. 代码示例
5. 边学边练
6. 易错点/注意事项
7. 本节挑战题
8. 总结与下一节引导

### 6.4 docx 到网站内容的映射规则

- `Heading 1` → `Lesson`
- `Heading 2/Heading 3` → lesson 内的分节标题
- 连续代码行 → `code` 或 `runner`
- “示例”段落 → 示例代码 + 运行器
- “语法/注意”段落 → 说明块 + 小测题源
- 图片 → media 资源，优先用于环境搭建、流程图、Linux 命令说明
- “课后作业” → 挑战题与题单

## 9. 关键交互要求

### 9.1 学习地图

- 节点按先修关系解锁。
- 每个节点展示：标题、难度、预计时长、完成状态、奖励。
- 完成 lesson 后即时弹出经验值与下一节点引导。

### 9.2 Lesson 学习体验

- 左侧/上方为讲解区，右侧/下方为代码区与输出区。
- “运行”和“提交”明确区分：
  - 运行：快速反馈，不计正式成绩。
  - 提交：进入隐藏用例判定，写入提交记录。
- 断线重连后可恢复 SSE 状态流。

### 9.3 移动端策略

- 允许浏览课程、查看题面、查看进度、签到。
- 对 Monaco 显示“PC 端体验更佳”的降级提示。
- 不把移动端完整编码体验作为 MVP 验收项。

### 9.4 趣味化交互

- WASM Demo 1：栈/堆与 `new/delete`
- WASM Demo 2：指针与引用
- Debug 题：提供一段错误代码，用户修复后运行/提交

## 10. 非功能要求

### 10.1 性能

- 课程页、题库页 P95 接口响应 < 300ms（不含判题结果）。
- `run` 请求应在健康队列下 5 秒内返回结果或明确排队状态。
- `submit` 必须异步化，1 秒内返回 `submissionId`。

### 10.2 安全

- 密码不明文存储。
- 判题默认禁网。
- 题目隐藏用例不可前端下发。
- 后台操作可审计。
- 提交接口、登录接口必须限流。

### 10.3 可运维性

- 关键链路可观测：登录、课程访问、提交、判题、SSE。
- 数据库具备日备份与恢复演练机制。
- Lesson 与 Problem 发布支持版本回滚。

## 11. 核心验收标准

### 11.1 内容验收

- 首发 29 节课全部可在后台发布。
- 每节课至少包含：正文、1 个示例代码块、1 个互动块、1 个验收点。
- 首发 120 题中，至少 20 题为正式判题题。

### 11.2 功能验收

- 用户能从首页进入路径并完成首条学习主线。
- 代码运行、正式提交、结果回写、SSE 推送完整可用。
- 个人中心能看到 XP、徽章、连续学习和提交记录。
- 后台能完成课程、题目、用例的新建与发布。

### 11.3 业务验收

- 新用户从注册到完成第一节课路径闭环可在 15 分钟内完成。
- 首发内容可以支撑“完整跑一条学习路径”的运营活动。

## 13. 结论

这个网站不应把自己做成“又一个在线 IDE”，而应做成“以 docx 课程资产为核心、以学习路径驱动、以运行和判题增强反馈、以成长系统增强留存”的 C++ 教学平台。MVP 先把 29 节核心课跑通，再按 `docs/product/course-map-from-docx.md:1` 的分组逐步扩展到完整体系。
