# Project Specification

## 1. 文档定位

本文件是本项目的 **执行级总规范**，用于统一定义项目在 AI / Agent 参与下的工作方式、文档分层、工件标准、固定模板和质量门禁。

它不是某一次需求的 PRD，也不是某一版技术方案的 Spec；它是位于这些业务文档之上的 **项目级上位约束文档**，用于回答以下问题：

- 项目中的文档应如何分层与引用；
- Agent / Skill 应如何接入项目并执行任务；
- 复杂任务应按照什么固定模板推进；
- 从输入、分析、计划、执行到发布、复盘，应遵循什么标准流程；
- 哪些规则属于必须遵守的全局执行要求。

自本文件生效后，`specification/` 目录以本文件作为唯一权威来源；后续凡涉及执行方式、模板结构、协作协议、质量门禁或项目级方法论调整，均应优先更新本文件，而不是在其它文档中分散定义。

本文件的主要读者与使用对象包括：

- 项目负责人：用于统一项目执行方式和协作边界；
- 产品经理：用于将需求输入接入固定执行模板；
- 架构师与研发负责人：用于约束技术方案、任务拆解和交付路径；
- 前后端工程师、QA、DevOps：用于理解工件要求、阶段门禁和协作协议；
- AI Agent / Codex / 多 Agent 系统：用于作为项目级默认执行规范和模板来源。

### 1.1 文档权威性

本文件在项目内部承担“执行方法与模板规则”的最高解释权。  
它不负责替代业务需求和技术设计本身，而是负责规定：

- 这些业务与技术文档如何进入执行流程；
- 执行过程中必须生成哪些工件；
- 各类工件之间的衔接关系；
- 哪些阶段必须通过门禁后才能进入下一阶段。

### 1.2 规则优先级

项目内规则优先级分为三层：

#### A. 执行边界优先级

从高到低：

1. 外部系统 / 平台 / 人类明确约束
2. 项目根目录 `AGENTS.md`
3. 本文件 `specification/project-specification.md`
4. `skills/*/SKILL.md`
5. 一次性任务说明或临时备注

#### B. 业务范围优先级

从高到低：

1. 经确认的 PRD
2. 执行计划
3. 任务拆解
4. 临时讨论记录

#### C. 技术实现优先级

从高到低：

1. 经确认的 Spec
2. 实施方案 / 技术说明
3. 任务拆解中的实现描述
4. 临时实现假设

如果不同层级发生冲突，应以上层规则为准；若同层文档冲突，必须进入 `待确认项`，不得自行裁决。

### 1.3 适用范围

本文件适用于以下场景：

- 新项目启动
- 新需求接入
- 执行计划生成
- 研发任务拆解
- 多 Agent 协作
- 发布与回滚准备
- 阶段复盘与资产沉淀

### 1.4 不适用范围

本文件不直接承担以下内容：

- 单个功能的业务细节定义
- 单个接口的字段级设计
- 单次发布的临时操作记录
- 单个 Bug 的局部修复说明

这些内容应分别沉淀到 PRD、Spec、API 文档、发布文档或问题记录中。

## 2. 核心结论

本项目应采用 **AI 原生、Agent 化、可约束、可审计、可复用** 的执行方式，而不是把 AI 当成一个临时问答工具。

其核心思想可以归纳为五条：

1. **先建护栏，再让 AI 开工**  
   不能在没有规则、没有检查、没有提交流水线的情况下让 AI 直接改项目。

2. **项目文档不是一份 PRD，而是一套执行环境**  
   必须通过“宪法层、技能层、知识层、状态层”形成可持续工作的上下文系统。

3. **把高频任务做成 Skill，而不是每次重写 Prompt**  
   API 设计、执行计划、代码审查、迁移设计、测试验证等都应被固化为可触发技能。

4. **多 Agent 协作必须有共享协议**  
   角色、路由、状态写入、工件产出、上下文交接都必须结构化。

5. **所有输出最终都要回到工程产物**  
   包括 PRD、Spec、任务计划、代码、日志、测试、部署、回滚和文档，而不是停留在聊天记录。

## 3. 项目执行总原则

### 3.1 边界优先

所有 AI 行为必须先受边界约束，再执行任务。边界分为三类：

- `ALWAYS`：总是必须执行的动作
- `ASK FIRST`：执行前必须征得确认的动作
- `NEVER`：绝对禁止的动作

这类边界应由项目级 `AGENTS.md` 承载。

### 3.2 文档分层

不要把所有规范堆在一个文件里。必须将项目执行上下文拆分为分层资产。

### 3.3 可验证优先

AI 的输出必须可检查、可运行、可复核。  
不能接受“逻辑上大概对”的结果，必须有：

- 结构校验
- 静态检查
- 测试结果
- 变更记录
- 工件落盘

### 3.4 最少幻觉原则

如果文档没有给出信息，不能自行发明。  
信息不足时必须：

- 标记为 `待确认项`
- 请求补充输入
- 或显式说明假设

### 3.5 资产化原则

一次性 Prompt 不应成为主要工作方式。  
成熟做法是将高频流程沉淀为：

- `AGENTS.md`
- `skills/*/SKILL.md`
- `docs/*.md`
- 模板
- 脚本
- 验证器

### 3.6 单一事实来源原则

每类信息应有单一事实来源：

- 业务目标与范围 → PRD
- 技术实现与约束 → Spec
- 项目执行方法 → Project Specification
- 单次项目推进节奏 → Execution Plan
- 任务责任与交付 → Task Breakdown
- 实时状态与阻塞 → Status Sync
- 发布动作与回滚 → Release / Rollback Plan

任何文档都不应越位成为其它类别信息的长期主来源。

### 3.7 模板驱动交付原则

当任务复杂度达到“跨角色、跨阶段、跨文档”任一条件时，必须模板化推进。

触发条件包括但不限于：

- 需要同时依赖 PRD 和 Spec
- 需要多人协同
- 需要阶段性评审
- 需要上线发布
- 需要后续复盘复用

一旦触发，就应使用本文件第 `13` 节中的固定模板，而不是依赖自由发挥的自然语言描述。

### 3.8 模板骨架自适配与自迭代原则

作为通用项目模板，本仓库的骨架必须满足以下能力：

1. **可初始化**：可以一键生成新项目基础结构；
2. **可适配**：可根据项目类型加载不同的目录、变量和默认文档；
3. **可同步**：模板演进后，可安全同步到已初始化项目；
4. **可追踪**：项目当前使用的模板状态、Profile 和变量必须可落盘；
5. **可保护**：同步模板时不得默认覆盖业务文档、原始资料和中间产物。

对应落地载体如下：

- `templates/template-manifest.json`：定义基础骨架与默认同步范围；
- `templates/profiles/*.json`：定义不同项目类型的默认差异；
- `templates/template-config.example.json`：定义项目级覆盖项；
- `templates/commands/bootstrap-project.ps1`：初始化新项目；
- `templates/commands/sync-template.ps1`：同步模板升级；
- `.template/template-state.json`：记录当前项目的模板状态。

## 4. 四层 Specification 架构

本项目的规范体系采用四层结构。

### 4.1 宪法层（Constitution）

载体：

- 项目根目录 `AGENTS.md`

作用：

- 定义项目边界
- 定义默认执行方式
- 定义必须先询问的高风险动作
- 定义禁区
- 定义常用命令和检查命令

要求：

- 内容简短、硬约束明确
- 使用 `ALWAYS / ASK FIRST / NEVER` 结构
- 只保留高价值规则，不写业务大段说明

### 4.2 技能层（Skills）

载体：

- `skills/*/SKILL.md`
- 可选的 `references/`、`scripts/`、`assets/`

作用：

- 将重复性任务标准化
- 为特定任务定义输入、步骤、输出格式和覆盖检查
- 避免每次重复解释流程

技能应覆盖的典型方向：

- 执行计划生成
- API 设计
- 数据模型设计
- 代码审查
- 变更风险评估
- 测试策略设计
- 发布与回滚准备

### 4.3 知识层（Knowledge）

载体：

- `docs/`
- 项目 PRD / Spec / 实施方案 / 运行约定等

作用：

- 作为项目权威知识源
- 被技能或 Agent 在执行前引用
- 承载业务规则、接口规则、结构规则、发布约束

知识层的原则：

- 不在多个文件里重复同一规则
- 统一命名
- 版本清晰
- 能被 Skill 引用

### 4.4 状态层（State）

载体可以是：

- 共享记忆
- 日志
- 执行记录
- 决策记录
- 待确认项清单
- 任务进度

作用：

- 支持跨会话延续
- 让多 Agent 能共享上下文
- 保留决策链路与历史状态

状态层原则：

- 记录事实，不记录冗余闲聊
- 记录可影响后续执行的上下文
- 强调当前状态、阻塞项、下一步动作

## 5. Day 0 护栏要求

在允许 AI 大规模参与开发前，必须完成项目基础护栏。

### 5.1 静态检查

对前端和后端都应开启严格检查。

建议：

- Web：TypeScript 严格模式
- Go：`go vet`、`golangci-lint`、格式化检查

### 5.2 Secret 扫描

必须阻止 AI 提交真实密钥、伪造密钥或敏感凭证。

建议：

- 使用 `detect-secrets` 或等价工具
- 在 commit 前执行扫描
- 将基线文件纳入项目

### 5.3 Pre-commit Hook

必须在本地阻止低质量变更进入仓库。

建议至少拦截：

- 格式化失败
- lint 失败
- 单元测试失败
- secret 扫描失败

### 5.4 高并发 CI

CI 不应串行阻塞整个团队。  
建议使用并行矩阵执行：

- lint
- build
- test
- format check
- security scan

### 5.5 工件保存

AI 的重要输出不能只留在对话里，必须落盘到项目目录，例如：

- PRD
- Spec
- 执行计划
- SQL
- OpenAPI
- 测试清单
- 发布文档

## 7. Skill 设计规范

### 7.1 Skill 的职责

Skill 是项目中的“可复用执行插件”，负责把复杂流程转为可重复执行的方法。

Skill 不是：

- 普通说明文
- 空泛原则集合
- 冗长背景介绍

Skill 应包含：

- 触发条件
- 必需输入
- 工作流
- 输出要求
- 覆盖检查
- 必要的参考资料

### 7.2 Skill 的输入规范

Skill 必须明确说明：

- 需要哪些输入
- 哪些输入缺失时不能继续
- 缺失时是自动定位、请求用户确认，还是进入 `待确认项`

### 7.3 Skill 的输出规范

Skill 应尽量输出：

- Markdown
- 结构化标题
- 表格化任务清单
- 可落盘的工件

### 7.4 Skill 的资源使用规范

Skill 内可包含：

- `references/`：上下文参考
- `scripts/`：可执行脚本
- `assets/`：模板或资源

优先级：

1. 能脚本化的尽量脚本化
2. 能模板化的尽量模板化
3. 正文只保留高价值执行知识

## 8. 多 Agent 协作规范

当任务复杂到需要多角色并行时，应使用多 Agent 协作。

### 8.1 典型角色

建议的项目角色包括：

- `Coordinator / Router`
- `PM / Analyst`
- `Architect`
- `Frontend Engineer`
- `Backend Engineer`
- `QA`
- `DevOps`
- `Content / Ops`

### 8.2 编排器（Coordinator）的职责

编排器只负责：

- 判断任务该交给哪个角色
- 决定是否需要先读取共享状态
- 决定是否需要生成工件
- 控制是否需要路由给多个 Agent 协同

编排器不应：

- 把内部路由过程直接暴露给用户
- 未经确认擅自扩大任务范围

### 8.3 角色 Agent 的职责

每个角色 Agent 应拥有：

- 明确身份
- 责任边界
- 输出风格
- 可用资料
- 不可越权的约束

### 8.4 共享协议

多 Agent 协作必须遵守共享协议：

- 共享输入文档
- 共享状态摘要
- 共享工件命名规范
- 共享待确认项列表
- 共享最终产出位置

## 9. Prompt 与执行协议

本项目不鼓励一次性长 Prompt 驱动所有行为。  
推荐采用“协议化 Prompt”。

### 9.1 输入协议

每个复杂任务开始前，应先确认：

- 目标
- 输入文档
- 输出工件
- 范围边界
- 技术约束
- 是否允许修改文件

### 9.2 分析协议

在真正执行前，先提取：

- 业务目标
- 功能范围
- 技术实现范围
- 文档冲突
- 风险
- 待确认项

### 9.3 输出协议

输出必须：

- 结构化
- 可读
- 可执行
- 可追溯到输入文档
- 能落盘

### 9.4 覆盖检查协议

所有交付物都应检查是否覆盖：

- 功能
- 接口
- 数据
- 配置
- 权限
- 日志
- 监控
- 测试
- 发布
- 回滚
- 文档更新

## 10. 文档治理规范

### 10.1 文档职责分离

不同文档只承担一种主要职责：

- PRD：产品需求与范围
- Spec：技术设计与实现边界
- Implementation：实施方案与技术路线
- PRD/Spec Execution Plan：执行计划与任务拆解
- Project Specification：项目方法论与执行总规范

### 10.2 避免重复

同一规则只应保留在一个权威文档中。  
其他文档通过引用或链接关系使用，不应复制粘贴多份版本。

### 10.3 文档更新原则

更新文档时应：

- 保留结构稳定
- 保证命名一致
- 同步相关工件
- 明确该文档是否为权威版本

## 11. 项目执行主工作流

本项目建议采用以下标准执行流程。

### Phase 0：环境与护栏

完成：

- 边界定义
- 静态检查
- 预提交校验
- CI
- 秘钥扫描

### Phase 1：输入确认

确认：

- PRD
- Spec
- 技术约束
- 时间范围
- 交付目标

### Phase 2：分析与映射

输出：

- 业务目标提取
- 功能模块提取
- 技术模块提取
- PRD ↔ Spec 映射
- 冲突和待确认项

### Phase 3：执行计划

输出：

- 任务拆解
- 角色分工
- 依赖关系
- 里程碑
- 风险分析

### Phase 4：设计与实现

输出：

- 详细设计
- API
- 数据结构
- 代码实现
- 配置
- 测试

### Phase 5：验证与上线

输出：

- 验证结果
- 监控方案
- 灰度策略
- 回滚方案
- 发布文档

### Phase 6：复盘与沉淀

输出：

- 经验总结
- 文档补全
- Skill 更新
- 状态记录

### 11.7 阶段进入 / 退出条件

| 阶段 | 进入条件 | 退出条件 |
| --- | --- | --- |
| Phase 0 环境与护栏 | 已明确项目目录与执行方式 | 边界、检查、CI、密钥扫描可用 |
| Phase 1 输入确认 | 已收到任务或需求输入 | PRD / Spec / 约束 / 目标已确认或列出缺口 |
| Phase 2 分析与映射 | 输入完整性检查已完成 | 完成业务、功能、技术和冲突提取 |
| Phase 3 执行计划 | 已形成 PRD / Spec 映射 | 已输出执行计划、依赖、里程碑、风险 |
| Phase 4 设计与实现 | 任务拆解已完成 | 实现、配置、测试和联调产物可验证 |
| Phase 5 验证与上线 | 发布前检查已启动 | 发布、监控、回滚方案和观察项齐备 |
| Phase 6 复盘与沉淀 | 里程碑或发布已结束 | 复盘、决策、模板与资产沉淀完成 |

## 13. 固定执行模板

本节将前述原则固化为一套固定执行模板。  
后续凡是涉及需求理解、技术设计、执行计划、任务拆解、阶段同步、发布上线、复盘沉淀，默认都应优先复用本节模板。

### 13.1 模板使用总则

1. 先确认输入，再产出模板。
2. 先做分析模板，再做执行模板。
3. 每个模板都必须能单独落盘。
4. 模板中没有信息的字段不得硬填。
5. 缺失信息统一进入 `待确认项`。
6. 每个模板都应尽量引用来源文档，而不是脱离输入独立发挥。
7. 若项目需要初始化或升级模板骨架，必须优先复用 `bootstrap-project.ps1` / `sync-template.ps1`，而不是手工散改目录结构。

### 13.2 固定执行顺序

标准执行顺序如下：

1. 任务受理单
2. 输入完整性检查
3. PRD / Spec 理解与映射
4. 执行计划
5. 详细任务拆解
6. 阶段状态同步
7. 发布与回滚清单
8. 复盘总结

### 13.3 固定工件命名约定

如无特殊说明，建议统一按以下命名方式落盘：

- `docs/{project}-prd.md`
- `docs/{project}-spec.md`
- `docs/{project}-execution-plan.md`
- `docs/{project}-task-breakdown.md`
- `docs/{project}-api-spec.md`
- `docs/{project}-data-model.md`
- `docs/{project}-test-plan.md`
- `docs/{project}-release-plan.md`
- `docs/{project}-rollback-plan.md`
- `docs/{project}-retrospective.md`

#### 13.3.1 最小工件集

对于一个完整可执行项目，最少应具备以下工件：

- PRD
- Spec
- Execution Plan
- Task Breakdown
- Status Sync（持续更新）
- Release / Rollback Plan（上线前）
- Retrospective（里程碑后）

如果缺少上述任一工件，应明确说明缺失原因和补齐时间。

#### 13.3.2 工件目录映射

建议按以下目录放置：

- `docs/`：正式业务、技术、执行、发布、复盘类工件
- `specification/`：项目级执行总规范
- `skills/`：可复用执行流程
- `generated/`：AI 草稿、中间版本、评审前产物
- `artifacts/` 或 `reports/`：导出报告、阶段性快照

#### 13.3.3 工件状态定义

统一使用以下状态：

- `draft`：草稿，尚未评审
- `in_review`：评审中
- `approved`：已确认，可作为后续输入
- `blocked`：因缺少输入或冲突暂不可推进
- `archived`：已归档，仅保留历史参考

#### 13.3.4 评审结论定义

统一使用以下结论：

- `pass`：通过，可进入下一阶段
- `pass_with_actions`：有遗留动作，但不阻塞下一阶段
- `reject`：不通过，需要回退修改
- `pending`：暂不决策，等待补充信息

### 13.14 模板字段填写规则

所有模板默认遵守以下填写规则：

1. 字段能引用来源时，优先引用来源文档。
2. 字段没有数据时，填 `待确认`、`不适用` 或留空后集中归入 `待确认项`，不得捏造。
3. 表格字段优先短句，不写大段散文。
4. 风险、依赖、阻塞项必须写“影响”，不能只写现象。
5. 每份模板都应能被独立阅读，不依赖聊天上下文才能理解。

#### 13.14.1 优先级字段定义

- `P0`：不完成则当前阶段无法成立，或直接影响上线 / 安全 / 核心链路
- `P1`：重要但可在 P0 后推进，会影响效率、质量或体验
- `P2`：增强项、补充项、可后置项

#### 13.14.2 复杂度字段定义

- `Low`：范围清晰、依赖少、实现路径稳定
- `Medium`：涉及多模块或中等依赖，需要联调或评审
- `High`：跨阶段、跨团队、跨系统，或存在明显不确定性

#### 13.14.3 状态字段定义

- `todo`：尚未开始
- `in_progress`：正在推进
- `blocked`：被依赖或待确认项阻塞
- `done`：已完成
- `verified`：已完成且通过验证

## 16. 反模式清单

以下做法在本项目中视为反模式：

- 只有一个大 Prompt，没有结构化文档资产
- 直接让 AI 自由改仓库，不设边界
- 任务依赖口头描述，不落盘
- 同一规则散落在多个文件中
- 文档与代码脱节
- 只生成方案，不生成任务拆解
- 只生成功能，不考虑监控、回滚、日志、权限和测试
- 输入不足时硬猜实现细节
